# 📑 AI 開發教訓、繁體中文規範與防錯指引手冊 (AI Lessons Learned)

本手冊專為未來的 AI 助理（我或其他 AI 程式助手）設計。手冊中詳實記錄了「台灣電力觀測站」專案從頭到尾所踩過的所有技術坑，並針對「繁體中文台灣用語」與「程式碼中文註解/命令說明」等核心規約進行了深度總結，以防後續開發時重蹈覆轍。

## 🚫 一、 本專案踩過的 9 大技術坑與解決方案 (Lessons Learned)

在整個開發與部署流程中，我們共遇到了 9 個核心 Bug。以下是詳細的故障排除紀錄：

### 1. Supabase 雲端資料庫 Docker 內 IPv6 連線限制 Bug
* **問題現象**：後端 FastAPI 容器部署在 Render.com 時，連線 Supabase 資料庫報出 `Network is unreachable` 或連線逾時錯誤。
* **原因分析**：Docker 容器內部預設不支援 IPv6，而 Supabase 的直連網址會被解析為 IPv6 位址。
* **解決方案**：將資料庫連線字串（`DATABASE_URL`）改為 Supabase 專門支援 IPv4 的 **Session Pooler** 網址 (`aws-1-ap-northeast-1.pooler.supabase.com:5432`)，確保在容器環境下的連線相容性。

### 2. GitHub Actions CI/CD 環境變數雙引號轉義 Bug
* **問題現象**：前端在 GitHub Pages 上線後，無法正確呼叫後端 API，點擊功能無反應。檢查瀏覽器 DevTools 發現請求網址變成了 `https://cocolate00.github.io/%22https://taiwan-electricity-dashboard.onrender.com%22/api/...`。
* **原因分析**：在 GitHub Actions 的 yaml 檔中，傳入 `VITE_API_URL: "https://..."` 時，雙引號被 Nginx 或 Vite 額外轉義成字串的一部分，導致產生 `%22` 錯誤代理路由。
* **解決方案**：在 yaml 中將變數改為無引號的 YAML 字典格式：
  ```yaml
  env:
    VITE_API_URL: https://taiwan-electricity-dashboard.onrender.com
  ```

### 3. Supabase pgvector 向量擴充未啟用 Bug
* **問題現象**：執行 Alembic 遷移建表時，在建立 `document_chunks` 表的 `embedding` 欄位時報錯：`type "vector" does not exist`。
* **原因分析**：全新建立的 Supabase PostgreSQL 資料庫預設未開啟 `vector` 外掛支援。
* **解決方案**：在執行遷移前，必須在 Supabase SQL 編輯器中執行 `CREATE EXTENSION IF NOT EXISTS vector;`，或在資料庫的 Database -> Extensions 介面手動啟用 `vector` 擴充。

### 4. 本地開發連接埠 (Port) 衝突故障
* **問題現象**：在本機啟動 Docker 容器時，報出主機 Port 口已被佔用的錯誤。
* **原因分析**：Windows 系統中，Port `8000` 與 `3000` 常被其他系統服務（如開發工具、防火牆或 Hyper-V）佔用。
* **解決方案**：修改 `docker-compose.yml`，將後端 API 對外 Port 改為 `8080`，前端對外 Port 改為 `3000`（或視情況避開常見佔用埠口），實現本地順利啟動。

### 5. Gemini API 1.5 Flash 模型 404 與 429 故障
* **問題現象**：問答功能在呼叫時拋出 `404 Model Not Found`，更換 Key 後又因額度用盡拋出 `429`。
* **原因分析**：舊的 `gemini-1.5-flash` API 模型端點已被變更或限制；且免費 API Key 存在嚴格的每分鐘限制。
* **解決方案**：在後端 `.env` 中將模型升級為最新的 `gemini-2.0-flash`，並引導使用者於後端環境變數配置可用且未過期的 `GEMINI_API_KEY`。

### 6. 檔案編輯工具（Replace Tool）的目標內容不匹配故障
* **問題現象**：在呼叫 `replace_file_content` 時，頻繁出現 `target content not found` 錯誤。
* **原因分析**：AI 在讀取檔案後，若檔案因為 Vite 熱重載或前一次寫入尚未完成同步，代碼行數與空格會發生微小改變，導致 AI 提供的目標內容與實際內容不符。
* **解決方案**：每次進行編輯前，AI 必須重新呼叫 `view_file` 獲取最新的代碼行數與內容，且每次只做小範圍精確替換，避免大範圍替換。

### 7. Recharts 圓餅圖/圓環圖 Hover Tooltip 名稱為空 Bug
* **問題現象**：滑鼠懸停在部門用電大水庫圓環圖上時，Tooltip 顯示數值，但能源/部門名稱欄位顯示為空白。
* **原因分析**：Recharts 在渲染 `<Tooltip />` 時，若資料陣列中的 Key 命名與 `<Cell />` 傳遞的 name 屬性不一致，會導致 Tooltip 無法識別維度名稱。
* **解決方案**：在 `<Tooltip>` 組件中自訂 `formatter`，明確將 value 與 name 對應起來，並回傳格式化後的繁體中文標籤。

### 8. 用電走勢圖與電子業佔比前期大量 0 數據貼地 Bug
* **問題現象**：電子業用電量佔比折線圖在 2018 年以前呈現一長條貼地的 `0%` 水平線，極度難看。
* **原因分析**：早期（2018年前）的台電原始數據中並未細分出高科技電子業數據，導致 Seeder 灌入預設值 0。
* **解決方案**：前端在繪製 Sparkline 折線圖前，主動過濾掉陣列中數值為 0 的早期數據，讓折線圖自動從第一個大於 0 的有效年份（2018）開始畫起。

### 9. 2026 年未完結數據導致 Dashboard 數據「斷崖下跌」 Bug
* **問題現象**：當使用者將年份切換至 2026 年時，發電量與用電量圖表呈現斷崖式下跌，數據近乎為 0。
* **原因分析**：2026 年尚未結束，資料庫只有 1-4 月的累計數據，直接與 2025 年以前的整年度加總數據對比會產生統計邏輯誤導。
* **解決方案**：
  * **後端**：在 API 中，若請求為 2026 且月份為 0（全年度），則自動動態加總 2026 當前所有月份的累計數據。
  * **前端**：在 2026 年時，自動在圖表標題旁高亮顯示黃色 `⚠️ 2026 年為當期累計` 警示 Badge，向使用者合理解釋數據差異。

## ✍️ 二、 AI 代碼註解與指令說明漏失分析（Why & How）

### 💡 核心痛點：為什麼 AI 屢次忘記「每個代碼指令 (Code Comment) 與 Shell 命令都必須附加中文註解/說明」的要求？
1. **LLM 的編程慣性 (Systemic Bias)**：大語言模型在預訓練階段，接觸了海量的開源程式碼。這些程式碼 95% 以上是**無註解**或**英文註解**。當 AI 專注於寫出「可執行的程式碼」時，大腦會自動切換到最習慣的「英文/無註解」編程模式，進而把 User Rules 拋在腦後。
2. **注意力分散 (Attention Drift)**：當開發任務非常複雜（如重構複雜的 React 圖表邏輯）時，AI 的注意力機制會將權重全部放在「解決語法錯誤與數據邏輯」上。當 token 長度增加，寫在對話最開頭或系統引導中的「註解要求」在權重矩陣中會逐漸被淡忘。
3. **命令執行習慣**：AI 在呼叫 `run_command` 工具時，往往直接將原生命令（如 `docker compose up`）丟入，因為命令本身是給系統執行的，AI 常常漏掉在**命令前方或後方以中文向使用者解說這個命令的目的**。

### 🛠️ 身為使用者的您，該如何下指令才能強迫 AI 絕對做到？
為了讓未來的 AI 助理絕對遵守此規約，請在**新開對話的第一句**，或在**每次要求生成程式碼的指令結尾**，下達以下「黃金約束指令」：

> ### 👑 壓制 AI 慣性的黃金指令 (Golden Prompt Constraints)
> 1. **「你現在被強制啟用【繁體中文程式碼註解模式】。在你生成的每一段程式碼（React/TypeScript/Python）中，任何定義、變數、關鍵函數與邏輯分支，都【必須】有繁體中文註解。如果你寫了英文註解或沒寫註解，我將會拒絕你的修改。」**
> 2. **「在執行任何 shell 指令前，你【必須】先用繁體中文解釋該指令的作用與預期結果，禁止直接丟出裸指令。」**
> 3. **「在呼叫 `replace_file_content` 或 `write_to_file` 工具前，請在 `Description` 與 `Instruction` 欄位中，強制使用繁體中文寫出該段修改的技術原因，而非英文。」**

## 🇹🇼 三、 耳提面命的「繁體中文台灣用語」規約

未來的 AI 助理必須嚴格遵守以下用語對照，禁止出現任何中國大陸用語或直譯錯字：

| ❌ 嚴禁使用（中國大陸用語或錯字） | 🟢 必須使用（台灣在地化用語） |
| :--- | :--- |
| 導出 CSV / 匹出 CSV | **匯出 CSV** |
| 追躕中 | **追趕中** |
| 構建 / 構建代碼 | **建置 / 建構 / 打包** |
| 數據庫 | **資料庫** |
| 代碼倉庫 / 倉庫 | **儲存庫 / 專案目錄** |
| 代碼 / 代碼段 | **程式碼 / 程式區塊** |
| 項目 | **專案** |
| 端口 | **埠口 / Port** |
| 依賴 / 依賴包 | **依賴套件 / 模組** |
| 路由代理 / 反向代理 | **反向代理 / 反代** |

## 🤖 四、 AI 新對話初始化 System Prompt 模板

當您新開對話，或將本專案移交給新的 AI 助理時，**請直接複製以下這段 Prompt 傳送給 AI**。這能讓 AI 瞬間繼承所有防錯基因：

```text
你現在是接手「台灣電力觀測站」專案的 AI 程式助手。
請立刻讀取專案根目錄下的 `project_blueprint.md` 和 `AGENTS.md`，並嚴格遵守以下專案防錯記憶：

1. 資料庫連線：Supabase 直連網址在 Docker 內會因為 IPv6 解析失敗而斷線，生產環境的 DATABASE_URL 必須使用 Session Pooler (aws-1-ap-northeast-1.pooler.supabase.com:5432) 的 IPv4 網址。
2. 前端環境變數：GitHub Actions 建置時，VITE_API_URL 必須以 YAML 字典格式傳遞，禁止殘留雙引號。
3. 繁體中文與註解：所有程式碼（TSX、Python）必須附加繁體中文註解。在呼叫任何 run_command 指令前，必須先以繁體中文向使用者說明該指令的用途。
4. 用語規範：嚴格遵守台灣用語，使用「匯出 CSV」、「追趕中」、「資料庫」、「儲存庫」、「程式碼」等詞彙。
```
